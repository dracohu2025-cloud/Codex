快速开始

在你的 IDE、CLI 或者云端直接开始使用 Codex。

1. Codex 桌面应用（macOS）

Codex 客户端目前专属提供给 macOS (Apple Silicon) 用户，为你提供一个独立的、悬浮于任何工作流之上的终极结对编程环境。

  1. 下载并安装： 运行并在弹出的窗口中通过 ChatGPT 或 OpenAI API Key 登录。
  2. 选择工作区： 挑选一个你需要 Codex 介入的本地工程文件夹。
  3. 发送你的第一条消息： 确保模式选定为 本地模式（让 Codex 原生运行在你机器上），然后随便问它关于代码的问题，比如："向我解释一下这个项目架构" 或 "在这个仓库下写一个贪吃蛇游戏"。

2. IDE 插件 (VS Code, Cursor, Windsurf)

让 Codex 常驻在你的代码编辑器侧边栏中。

  1. 在插件市场搜索并下载 Codex / ChatGPT 官方插件。
  2. 打开侧边栏面板： 登录后，Codex 默认会以 智能体模式 启动。这意味着它不仅能对话，更能自主阅读文件、在你的终端跑指令，以及直接拉起 Diff 试图修改你的代码。
  3. 建立 Git 快照： 既然 Codex 有能力直接改写代码，我们强烈建议你在交办任务前后提交 Git 记录，就像打游戏时的存档点，以便随时撤销误操作。

3. 命令行工具

这是开放给所有终端黑客（macOS, Windows, Linux 全平台）的最原生体验：

npm install -g @openai/codex
# 或者使用 Homebrew 安装
brew install codex

安装完成后，在所在的项目目录下敲入 codex 即可唤醒它。验证身份后，直接告诉它当前目录要完成什么任务。

4. 云端平台

不消耗任何本地算力？你可以前往网页版 chatgpt.com/codex 开辟云端战场。

  • 连接仓库： 按照向导关联你的 GitHub 仓库。只要环境跑起来了，哪怕页面关掉它也会在后台持续执行任务。
  • 审查并下发拉取请求： 当它完成任务后，你可以直接在云端看 Diff。如果觉得一切完美就点击「创建拉取请求」。 当然你也可用 git fetch 与 git checkout <branch-name> 拉回本地跑一遍测试。

应用功能

使用 Codex 应用可以做什么。

Codex 应用是一个专注的桌面体验，用于并行处理 Codex 线程，具有内置的工作树支持、自动化和 Git 功能。

跨项目多任务处理

使用一个 Codex 应用窗口跨项目运行任务。为每个代码库添加一个项目，并根据需要在它们之间切换。

如果你使用过 Codex CLI，项目就像在特定目录中启动会话。

如果你在一个仓库中处理两个或更多应用或包，将不同的项目拆分为单独的应用项目，以便沙盒仅包含该项目的文件。

技能支持

Codex 应用支持与 CLI 和 IDE 扩展相同的代理技能。你还可以通过点击侧边栏中的"技能"查看和探索团队在不同项目中创建的新技能。

自动化

你可以将技能与自动化相结合，执行例行任务，例如评估遥测中的错误并提交修复或创建有关最近代码库更改的报告。

模式

每个线程在选定的模式下运行。启动线程时，你可以选择：

  • 本地：直接在你当前的项目目录中工作。
  • 工作树：在 Git 工作树中隔离更改。
  • 云端：在配置的云端环境中远程运行。

本地和工作树线程都将在你的计算机上运行。

内置 Git 工具

Codex 应用直接在应用内提供常见的 Git 功能。

差异窗格显示本地项目或工作树检出中更改的 Git 差异。你还可以添加内联评论供 Codex 处理，并暂存或恢复特定的块或整个文件。

你还可以直接从 Codex 应用内为本地和工作树任务提交、推送和创建拉取请求。

对于更高级的 Git 任务，请使用集成终端。

工作树支持

创建新线程时，选择本地或工作树。本地直接在你的项目中工作。工作树创建一个新的 Git 工作树，以便更改与你的常规项目保持隔离。

当你想尝试新想法而不触碰当前工作，或者希望 Codex 在同一项目中并排运行独立任务时，请使用工作树。

自动化在 Git 仓库的专用后台工作树中运行，在非版本控制项目的项目目录中直接运行。

集成终端

每个线程都包含一个内置终端，范围限定为当前项目或工作树。使用应用右上角的终端图标或按 Cmd+J 切换它。

使用终端验证更改、运行脚本和执行 Git 操作，而无需离开应用。

常见任务包括：

  • git status
  • git pull --rebase
  • pnpm test 或 npm test
  • pnpm run lint 或类似的项目命令

如果你定期运行任务，可以在本地环境中定义一个操作，以在 Codex 应用窗口顶部添加一个快捷按钮。

注意，Cmd+K 在 Codex 应用中打开命令面板。它不会清除终端。要清除终端，请使用 Ctrl+L。

语音听写

使用语音提示 Codex。当编辑器可见时按住 Ctrl+M 并开始说话。你的语音将被转录。编辑转录的提示或点击发送让 Codex 开始工作。

浮动弹出窗口

将活动对话线程弹出到一个单独的窗口，并将其移动到正在积极工作的地方。这非常适合前端工作，你可以将线程保留在浏览器、编辑器或设计预览附近，同时快速迭代。

你还可以切换弹出窗口以保持置顶，当你希望它在整个工作流中保持可见时。

与 IDE 扩展同步

如果你在编辑器中安装了 Codex IDE 扩展，当你处于同一项目时，你的 Codex 应用和 IDE 扩展会自动同步。

同步时，你会在 Codex 应用编辑器中看到 IDE 上下文选项。启用"自动上下文"后，Codex 应用会跟踪你正在查看的文件，因此你可以间接引用它们（例如，"这个文件是关于什么的？"）。你还可以在 IDE 扩展中看到 Codex 应用中运行的线程，反之亦然。

如果你不确定应用是否包含上下文，请将其关闭并再次询问相同的问题以比较结果。

审批和沙盒

你的审批和沙盒设置约束 Codex 操作。

  • 审批决定 Codex 在运行命令之前何时暂停请求权限。
  • 沙盒控制 Codex 可以使用的目录和网络访问。

当你看到"批准一次"或"批准本次会话"等提示时，你正在为工具执行授予不同范围的权限。如果你不确定，请批准最窄的选项并继续迭代。

默认情况下，Codex 将工作范围限定为当前项目。在大多数情况下，这是正确的约束。

如果你的任务需要跨多个仓库或目录工作，请优先打开单独的项目或使用工作树，而不是要求 Codex 在项目根目录之外漫游。

MCP 支持

Codex 应用、CLI 和 IDE 扩展共享 Model Context Protocol (MCP) 设置。如果你已在其中一个中配置了 MCP 服务器，它们会自动被其他服务器采用。要配置新服务器，请打开应用设置中的 MCP 部分，然后启用推荐的服务器或向配置中添加新服务器。

网页搜索

Codex 附带一个第一方网页搜索工具。对于 Codex IDE 扩展中的本地任务，Codex 默认启用网页搜索，并从网页搜索缓存中提供结果。如果你将沙盒配置为完全访问，网页搜索默认为实时结果。请参阅配置基础以禁用网页搜索或切换到获取最新数据的实时结果。

图像输入

你可以将图像拖放到提示编辑器中以将它们作为上下文包含在内。拖放图像时按住 Shift 键以将图像添加到上下文。

你还可以要求 Codex 查看系统上的图像。通过为 Codex 提供截取你正在处理的应用屏幕截图的工具，Codex 可以验证它所做的工作。

通知

默认情况下，当应用在后台时，Codex 应用会在任务完成或需要批准时发送通知。

在 Codex 应用设置中，你可以选择从不发送通知或始终在应用处于焦点时发送通知。

保持计算机唤醒

由于你的任务可能需要一段时间才能完成，你可以让 Codex 应用通过启用应用设置中的"运行时防止睡眠"开关来防止计算机进入睡眠状态。

提示词指南

在开发语境下，如何高效地"点拨" Codex 智能体？

提示词：面向交付的工程指令

Codex 并不是普通的聊天机器人，你发出去的每一条问题或需求都将被转化为"循环指令"。比如 "写一个脚本输出 JSON 日志"。一旦给出，Codex 开启它自己的 读取-修改-执行循环：自我检索模型知识库 -> 读取本地目录里的现有代码 -> 修改或调用工具 -> 评估结果 -> 自我纠正。这个进程只有等它觉得完成了（或是被你大喊"停下"）才会终止。

高分秘籍：为 AI 创造一个可证伪的环境
Codex 在能够自己"验证结论"时产出最具震撼力。在布置任务的同时，给它一个具体的验收标准。比如丢几行 pnpm test 或是报错复现的 Terminal 日志给它，然后说："写这段代码，然后确保你的改动通过这个测试才算完。" 它接下来就真的会进入 TDD（测试驱动开发）模式为你全自动干活！

化繁为简：大模块切分

它能干很牛的事，但要分解。处理超巨型需求（比如"帮我把项目里遗留的 GraphQL 彻底替换成 REST"）时，最好把任务打散成小线程（Threads）。小任务意味着更小更清晰的审查（Review）范围。如果你根本不知道咋拆，你甚至可以直接把超巨型需求喂给它，然后说："为以上目标给我制定一个 5 步实施方案"，让它帮你规划。

线程与上下文压缩

每个对话都是一个独立的 线程，尽量不要同时派两个智能体跑两个线程 去修改同一份核心配置文件，这会像两个没通气的程序员一样引发冲突。
在处理非常漫长的马拉松编码任务时，大模型必然面临 Token 容量问题。不用担心，Codex 的架构内部实现了一种称为 上下文压缩（自动无损压缩） 的神技：当历史对话太厚时，它会自动在后台把老旧无用的试错流水账提取、缩写为关键逻辑状态，确保在几百轮对话后它的智商依旧在线，并且完全记得这栋大楼的地基在哪里。

核心功能

1. 智能体模式（默认启用）

在此模式下，Codex 拥有完整的"读 - 写 - 执行"权限：

  • 读取文件： 自动打开并分析你打开的任何文件
  • 编辑代码： 直接生成 Diff 并应用到你当前编辑的文件
  • 运行终端命令： 可以执行 npm install、git commit 等命令
  • 创建/删除文件： 根据需求生成全新的文件或清理无用文件

2. 内联编辑

选中一段代码，右键选择 "Codex: 编辑选中内容"，然后输入你的修改指令：

# 示例指令
- "把这个函数改成 async/await 风格"
- "添加错误处理逻辑"
- "提取这段重复代码为一个独立函数"

3. 代码解释

配置项详解

模型设置

  • model.name - 选择使用的模型版本
  • model.temperature - 创造性程度 (0-1)，越低越保守
  • model.maxTokens - 单次响应的最大 Token 数

行为设置

  • behavior.autoApprove - 是否自动应用更改（生产环境建议关闭）
  • behavior.previewChanges - 应用前显示 Diff 预览
  • behavior.confirmBeforeExecute - 执行这些命令前需要二次确认

上下文设置

  • context.excludePatterns - 忽略这些文件（支持 glob 语法）
  • context.includePatterns - 只读取这些类型的文件

MCP 协议

模型上下文协议 - 让 Codex 连接外部世界。

什么是 MCP？

Model Context Protocol (MCP) 是一个开放标准，允许 AI 模型（如 Codex）与外部数据源和工具进行标准化通信。通过 MCP，Codex 可以：

  • 查询数据库获取实时数据
  • 调用第三方 API（如 Stripe、Supabase、Linear）
  • 访问企业内部系统
  • 执行专门的工具链操作

架构概览

┌─────────────┐      ┌──────────────┐      ┌─────────────┐
│   Codex     │◄────►│  MCP服务器   │◄────►│  外部服务   │
│  (Client)   │      │  (Bridge)    │      │  (API/DB)   │
└─────────────┘      └──────────────┘      └─────────────┘

多智能体系统允许你同时运行多个 Codex 实例，每个实例负责不同的子任务。通过合理分工，可以大幅提升复杂任务的执行效率。

  • 代码审查：多个智能体从不同角度审查代码（安全性、性能、可维护性）

恢复对话

Codex 在本地存储你的转录，因此你可以接着离开的地方继续，而无需重复上下文。当你想要重新打开具有相同仓库状态和指令的先前线程时，使用 resume 子命令。

  • codex resume 启动最近交互式会话的选择器。高亮显示运行以查看其摘要，然后按 Enter 重新打开。
  • codex resume --all 显示超出当前工作目录的会话，因此你可以重新打开任何本地运行。
  • codex resume --last 跳过选择器，直接进入当前工作目录的最近会话（添加 --all 以忽略当前工作目录过滤器）。
  • codex resume <SESSION_ID> 针对特定运行。你可以从选择器、/status 或 ~/.codex/sessions/ 下的文件中复制 ID。

非交互式自动化运行也可以恢复：

codex exec resume --last "Fix the race conditions you found"
codex exec resume 7f9f9a2e-1b3c-4c7a-9b0e-.... "Implement the plan"

每个恢复的运行保留原始转录、计划历史和批准，因此 Codex 可以在你提供新指令时使用先前的上下文。

运行本地代码审查

在 CLI 中输入 /review 打开 Codex 的审查预设。CLI 启动一个专用的审查器，读取你选择的差异，并报告优先的、可操作的发现，而无需触碰你的工作树。

  • Review against a base branch：选择本地分支；Codex 找到与其上游的合并基础，对比你的工作，并在你打开拉取请求之前高亮显示最大的风险。
  • Review uncommitted changes：检查所有已暂存、未暂存或未跟踪的内容，以便你可以在提交之前解决问题。
  • Review a commit：列出最近的提交，并让 Codex 读取你选择的 SHA 的确切变更集。
  • Custom review instructions：接受你自己的措辞（例如"Focus on accessibility regressions"），并使用该提示运行相同的审查器。

Codex 附带一个第一方网页搜索工具。对于 Codex CLI 中的本地任务，Codex 默认启用网页搜索，并从网页搜索缓存中提供结果。缓存是 OpenAI 维护的网页结果索引，因此缓存模式返回预索引的结果，而不是获取实时页面。这减少了来自任意实时内容的提示注入风险，但你仍应将网页结果视为不受信任的。如果你使用 --yolo 或其他完全访问沙盒设置，网页搜索默认为实时结果。

使用输入提示运行

当你只需要快速回答时，使用单个提示运行 Codex 并跳过交互式 UI。

codex "explain this codebase"

Codex 将读取工作目录，制定计划，并在退出前将响应流式传输回你的终端。将其与 --path 等标志配对以针对特定目录，或使用 --model 预先调整行为。

Shell 补全

通过为 shell 安装生成的补全脚本来加快日常使用：

在 shell 配置文件中运行补全脚本以设置新会话的补全。例如，如果你使用 zsh，可以将以下内容添加到 ~/.zshrc 文件的末尾：

审批模式

审批模式定义 Codex 在停止确认之前可以执行多少操作。在交互式会话中使用 /permissions 切换模式。

  • Auto (default)：允许 Codex 读取文件、编辑并在工作目录中运行命令。在触碰该范围之外的任何内容或使用网络之前，它仍会询问。
  • Read-only：将 Codex 保持在咨询模式。它可以浏览文件，但在你批准计划之前不会进行更改或运行命令。
  • 完全访问：授予 Codex 在你的机器上工作的能力，包括网络访问，而无需询问。仅在信任仓库和任务时谨慎使用。

脚本化 Codex

使用 exec 子命令自动化工作流或将 Codex 连接到你现有的脚本中。这将以非交互方式运行 Codex，将最终计划和结果传回 stdout。

codex exec "fix the CI failure"

将 exec 与 shell 脚本结合使用以构建自定义工作流，例如自动更新变更日志、分类问题或在 PR 发布前执行编辑检查。

与 Codex Cloud 一起工作

codex cloud 命令让你无需离开终端即可分类和启动 Codex Cloud 任务。不带参数运行它以打开交互式选择器，浏览活动或已完成的任务，并将更改应用到本地项目。

提示编辑器

当你起草较长的提示时，切换到完整编辑器然后发送结果回编辑器可能会更容易。

在提示输入中，按 Ctrl+G 打开由 VISUAL 环境变量（如果未设置 VISUAL，则为 EDITOR）定义的编辑器。

技巧和快捷方式

  • 在编辑器中输入 @ 打开工作区根目录的模糊文件搜索；按 Tab 或 Enter 将高亮显示的路径放入你的消息中。
  • Codex 运行时按 Enter 将新指令注入当前轮次，或按 Tab 为下一轮次排队跟进提示。
  • 在行首使用 ! 运行本地 shell 命令（例如 !ls）。Codex 将输出视为用户提供命令结果，并仍应用你的审批和沙盒设置。
  • 编辑器为空时点击 Esc 两次以编辑你之前的用户消息。继续按 Esc 以在转录中进一步回退，然后按 Enter 从该点分支。
  • 使用 codex --cd <path> 从任何目录启动 Codex 以设置工作根目录，而无需先运行 cd。活动路径显示在 TUI 标题中。

非交互模式

使用 codex exec 在脚本和 CI 中运行 Codex。

非交互模式让你可以从脚本（例如持续集成 (CI) 作业）运行 Codex，而无需打开交互式 TUI。使用 codex exec 调用它。

何时使用 codex exec

在以下情况下使用 codex exec：

  • 作为管道的一部分运行（CI、合并前检查、计划作业）。
  • 生成可以管道传输到其他工具的输出（例如生成发布说明或摘要）。
  • 使用显式、预设的沙盒和审批设置运行。

基本用法

将任务提示作为单个参数传递：

codex exec "summarize the repository structure and list the top 5 risky areas"

当 codex exec 运行时，Codex 将进度流式传输到 stderr，并仅将最终代理消息打印到 stdout。这样可以轻松重定向或管道传输最终结果：

codex exec "generate release notes for the last 10 commits" | tee release-notes.md

当你不想将会话展开文件持久化到磁盘时，使用 --ephemeral：

codex exec --ephemeral "triage this repository and suggest next steps"

权限和安全

默认情况下，codex exec 在只读沙盒中运行。在自动化中，设置工作流所需的最低权限：

# 允许编辑
allow edits: codex exec --full-auto ""

# 允许更广泛的访问
allow broader access: codex exec --sandbox danger-full-access ""

仅在受控环境（例如隔离的 CI 运行器或容器）中使用 danger-full-access。

使输出机器可读

要在脚本中使用 Codex 输出，请使用 JSON Lines 输出：

codex exec --json "summarize the repo structure" | jq

当你启用 --json 时，stdout 变为 JSON Lines (JSONL) 流，因此你可以捕获 Codex 在运行时发出的每个事件。事件类型包括 thread.started、turn.started、turn.completed、turn.failed、item.* 和 error。

项目类型包括代理消息、推理、命令执行、文件更改、MCP 工具调用、网页搜索和计划更新。

示例 JSON 流（每行是一个 JSON 对象）：

{"type":"thread.started","thread_id":"0199a213-81c0-7800-8aa1-bbab2a035a53"}
{"type":"turn.started"}
{"type":"item.started","item":{"id":"item_1","type":"command_execution","command":"bash -lc ls","status":"in_progress"}}
{"type":"item.completed","item":{"id":"item_3","type":"agent_message","text":"Repo contains docs, sdk, and examples directories."}}
{"type":"turn.completed","usage":{"input_tokens":24763,"cached_input_tokens":24448,"output_tokens":122}}

如果你只需要最终消息，请使用 -o <path> 或 --output-last-message <path> 将其写入文件。这会将最终消息写入文件，同时仍将其打印到 stdout。

使用模式创建结构化输出

如果你需要为下游步骤提供结构化数据，请使用 --output-schema 请求符合 JSON Schema 的最终响应。这对于需要稳定字段的自动化工作流（例如作业摘要、风险报告或发布元数据）非常有用。

schema.json 示例：

{
  "type": "object",
  "properties": {
    "project_name": { "type": "string" },
    "programming_languages": {
      "type": "array",
      "items": { "type": "string" }
    }
  },
  "required": ["project_name", "programming_languages"],
  "additionalProperties": false
}

使用模式运行 Codex 并将最终 JSON 响应写入磁盘：

codex exec "Extract project metadata" \
  --output-schema ./schema.json \
  -o ./project-metadata.json

示例最终输出：

{
  "project_name": "Codex CLI",
  "programming_languages": ["Rust", "TypeScript", "Shell"]
}

在 CI 中进行身份验证

codex exec 默认重用保存的 CLI 身份验证。在 CI 中，通常需要显式提供凭据：

  • 将 CODEX_API_KEY 设置为作业的秘密环境变量。
  • 记住提示和工具输出可能包含敏感代码或数据。

常见自动化模式

示例：在 GitHub Actions 中自动修复 CI 故障

你可以使用 codex exec 在 CI 工作流失败时自动提议修复。典型模式是：

  1. 当主 CI 工作流以错误完成时触发后续工作流。
  2. 检出失败的提交 SHA。
  3. 安装依赖项并使用狭窄的提示和最小权限运行 Codex。
  4. 重新运行测试命令。
  5. 使用生成的补丁打开拉取请求。

使用 Codex CLI 的最小工作流

以下示例显示核心步骤。调整安装和测试命令以匹配你的技术栈。

name: Codex auto-fix on CI failure

on:
  workflow_run:
    workflows: ["CI"]
    types: [completed]

permissions:
  contents: write
  pull-requests: write

jobs:
  auto-fix:
    if: ${{ github.event.workflow_run.conclusion == 'failure' }}
    runs-on: ubuntu-latest
    env:
      OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
      FAILED_HEAD_SHA: ${{ github.event.workflow_run.head_sha }}
      FAILED_HEAD_BRANCH: ${{ github.event.workflow_run.head_branch }}
    steps:
      - uses: actions/checkout@v4
        with:
          ref: ${{ env.FAILED_HEAD_SHA }}
          fetch-depth: 0

      - uses: actions/setup-node@v4
        with:
          node-version: "20"

      - name: Install dependencies
        run: |
          if [ -f package-lock.json ]; then npm ci; else npm i; fi

      - name: Install Codex
        run: npm i -g @openai/codex

      - name: Run Codex
        run: |
          codex exec --full-auto --sandbox workspace-write \
            "Read the repository, run the test suite, identify the minimal change needed to make all tests pass, implement only that change, and stop. Do not refactor unrelated files."

      - name: Verify tests
        run: npm test --silent

      - name: Create pull request
        if: success()
        uses: peter-evans/create-pull-request@v6
        with:
          branch: codex/auto-fix-${{ github.event.workflow_run.run_id }}
          base: ${{ env.FAILED_HEAD_BRANCH }}
          title: "Auto-fix failing CI via Codex"

GitHub 自动化（Action）

在 CI/CD 流水线中使用 Codex。

安装 Action

# .github/workflows/codex.yml
name: Codex Auto Tasks
on:
  pull_request:
    types: [opened, synchronize]
  issues:
    types: [opened]

jobs:
  codex:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Run Codex
        uses: openai/codex-action@v1
        with:
          api-key: ${{ secrets.OPENAI_API_KEY }}
          task: "review-code"
          fail-on-issues: true

可用任务

  • review-code - 代码审查
  • fix-lint - 自动修复 Lint 错误
  • generate-tests - 生成单元测试
  • update-docs - 更新文档

配置选项

with:
  api-key: ${{ secrets.OPENAI_API_KEY }}
  model: codex-latest
  task: review-code
  working-directory: ./src
  fail-on-issues: true
  create-comments: true

功能成熟度

理解 Codex 文档中的成熟度标签及其含义。

部分 Codex 功能会标注成熟度，用于说明其稳定性、变更风险与支持等级。

成熟度 含义 建议
开发中（Under development） 尚未准备好使用。 不要使用。
实验性（Experimental） 不稳定，OpenAI 可能调整或移除。 自行承担风险。
测试版（Beta） 功能基本完整，可广泛测试，但仍可能根据反馈做调整。 适合评估与试点。
稳定版（Stable） 完整支持并文档化，行为与配置相对稳定。 可用于生产环境。
